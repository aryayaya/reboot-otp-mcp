# Security

本文说明 reboot-otp 流程的安全模型。

这个项目要缓解的是一个非常具体的风险：

- 在 Telegram / chat 驱动的操作流程里暴露长期有效的 Linux 密码

它通过以下方式来降低该风险：

- 用短时效 6 位 TOTP 替代长期密码输入
- 在主机本地完成 OTP 校验
- 只允许一个极窄的高权限动作

这是一种面向特定工作流的安全加固手段，不是完整的权限管理系统。

---

## 1. 安全目标

这个方案的安全目标是：

- 避免在聊天链路中传输长期 Linux 密码
- 为单一高权限动作增加短时效 OTP 门禁
- 把高权限动作固定为一个可审计的精确命令
- 不放宽通用 exec/shell 的默认限制

在本项目中，这个动作明确固定为：

```text
sudo /usr/sbin/reboot
```

---

## 2. 本项目缓解了什么风险

### 2.1 聊天链路中的长期密码暴露

如果原本的运维流程需要在 Telegram 或 Bot 对话里输入 Linux 密码，那么密码可能暴露于：

- 聊天记录
- 截图
- 复制粘贴历史
- bot 基础设施
- 人为误发
- 日志或转录内容

长期密码一旦泄漏，其危害通常比一次性 OTP 更大。

### 2.2 长期密码的可重复利用风险

相比之下，TOTP code 的特点是：

- 有时间窗口限制
- 不适合长期复用
- 暴露后的使用价值更短暂

### 2.3 高权限动作范围收敛

这个设计不是开放通用 passwordless sudo，而是把动作固定为一个精确命令。

---

## 3. 本项目不能防什么

这个项目**不能**覆盖所有风险。

### 3.1 不能防 Telegram 账号被盗

如果攻击者控制了被允许下达命令的 Telegram 账号，那么他仍可能发起重启请求。

### 3.2 不能防 TOTP secret 泄漏

如果 `TOTP_SECRET` 从主机、备份、截图、笔记或其他不安全位置泄漏，攻击者就可以生成有效 OTP。

### 3.3 不能防主机已经失陷

如果主机本身已经被控制，攻击者可能：

- 读取 secret 文件
- 读取进程环境变量
- 替换 MCP binary
- 绕过 OTP 检查
- 直接触发高权限路径

### 3.4 不能让宽泛 sudoers 变安全

本方案依赖的是**极窄** sudoers 规则。

### 3.5 不能替代 Telegram / Picoclaw 既有身份控制

这个仓库假定 Picoclaw 侧已有基本 sender 控制或授权边界。

---

## 4. 信任边界

### 被假定为可信的部分

- 本地 Linux 主机
- 本地 Picoclaw runtime
- 本地 Go MCP server binary
- sudoers 配置
- 存放 `TOTP_SECRET` 的本地 secret 文件
- 你的认证器设备

### 暴露面更高的部分

- Telegram / chat 传输链路
- 用户能看到的消息内容
- 运维日志 / 截图 / 转录
- 复制粘贴文本

本项目存在的意义，就是尽量不要把长期 Linux 密码放到这些暴露面更高的层里。

---

## 5. 设计的安全属性

### 5.1 本地校验

OTP 在主机本地校验，而不是交给远端服务。

### 5.2 窄命令面

被允许的高权限动作固定为：

```text
/usr/sbin/reboot
```

### 5.3 短时效授权值

这里使用的是 time-based OTP，而不是可长期复用的密码。

### 5.4 分层职责

整条链路被分成：

- skill：限制命令形状
- MCP server：本地校验 OTP
- sudoers：只放行一个精确命令

---

## 6. 为什么范围要故意保持很窄

本仓库故意只支持：

```text
/reboot --otp 123456
```

以及对应的唯一高权限动作：

```text
sudo /usr/sbin/reboot
```

这种“窄范围”本身就是安全特性。

它避免了：

- 任意命令执行
- shell 元字符处理
- 自然语言歧义
- 参数扩展导致的权限漂移
- 为了方便而不断扩大 sudo 边界

---

## 7. Secret 处理要求

`TOTP_SECRET` 是长期敏感信息，必须像真正的凭据一样保管。

### 要求

- 放在仓库之外
- 不进入 git
- 文件权限严格
- 不出现在聊天、日志、截图、笔记中
- 一旦怀疑泄漏就立即轮换

### 推荐存放方式

推荐放在类似：

```text
~/.picoclaw/secrets/privileged.env
```

的私有 env 文件中。

---

## 8. OTP 处理要求

一次性的 OTP 虽然比长期 secret 风险低，但仍然是敏感的临时授权值。

### 要求

- 不要在响应中回显 OTP
- 不要在日志里记录 OTP
- 不要无意义地持久化 OTP
- 只接受预期的固定格式

---

## 9. 日志与响应安全

安全的响应示例：

- `Usage: /reboot --otp 123456`
- `OTP verification failed. Reboot request denied.`
- `OTP verification is not configured.`
- `OTP verified. System is rebooting.`

不应该做的事：

- 回显用户提交的 OTP
- 打印 `TOTP_SECRET`
- 把包含 OTP 的完整命令文本直接写进日志
- 把内部堆栈细节原样暴露给聊天用户

---

## 10. Sudoers 安全边界

sudoers 是整条设计里最关键的边界之一。

它应当只允许：

- 一个用户
- 一个精确命令路径
- 不需要密码
- 且仅此而已

危险的放宽包括：

- `NOPASSWD: ALL`
- wildcard command path
- shell access
- 一大类系统命令

---

## 11. Telegram 场景特有的考虑

它避免了把长期 Linux 密码发进聊天链路。

但它并没有把 Telegram 变成一个天然安全的运维控制面。

风险仍包括：

- 账号被盗
- 设备被盗
- 消息历史泄漏
- 截图传播
- 操作失误

---

## 12. 主机失陷时的局限性

如果主机已经失陷，这套设计应视为退化。

攻击者在足够主机权限下，可能做到：

- 读取 secret 文件
- 查看进程环境
- 替换 MCP binary
- 去掉 OTP 验证逻辑
- 直接触发 reboot 路径

---

## 13. 运维建议

为了让实际部署尽量符合安全目标，建议：

- 保持只做一个精确动作
- secret 不进 git
- 文件权限尽量收紧
- 不要使用宽泛 sudoers
- 路径全部使用绝对路径
- 先做 harmless mode 验证
- 确保日志不含 OTP 或 secret

如果以后确实需要更多高权限动作，也建议按“一个动作一个审查”的方式逐步增加。

---

## 14. 已知限制

已知限制包括：

- 该动作仍然可以通过已授权的远程聊天链路触发
- 可用性依赖本地主机、Picoclaw、MCP server 都工作正常
- 安全性高度依赖 `TOTP_SECRET` 的保护质量
- 一次成功请求会导致真实系统状态变化：重启

---

## 15. 非目标

这个仓库**不打算**提供：

- 通用 OTP root shell
- 任意高权限命令执行
- 替代主机加固措施
- 替代 Telegram 账号安全
- 替代 sudo policy 设计
- 替代完整远程管理系统
